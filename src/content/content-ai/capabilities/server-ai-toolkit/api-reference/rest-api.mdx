## 认证

所有请求都必须包含：

- `Authorization: Bearer <JWT>`
- `X-App-Id: <APP_ID>`

请在服务端使用你的 AI 密钥生成 JWT。请在 [Server AI
Toolkit 设置](https://cloud.tiptap.dev/v2/cloud/ai) 页面获取你的 App ID 和密钥。

```ts
const JWT_TOKEN = jwt.sign(
  {
    experimental_document_server_id: 'your-document-server-id',
    experimental_document_server_management_api_secret:
      'your-document-server-management-api-secret',
  },
  'your-ai-secret-key',
  { expiresIn: '1h' },
)
```

有关更多信息，请参阅 [授权指南](/content-ai/capabilities/server-ai-toolkit/install#set-up-authorization)。

### Document Server 凭据

当你希望 Server AI Toolkit 自动获取并保存 Tiptap Cloud 文档时，请在 JWT 声明中包含这些 claims：

- `experimental_document_server_id`
- `experimental_document_server_management_api_secret`

## 如何提供文档

托管的基础 URL 是：

```txt
https://api.tiptap.dev
```

在下面的示例中，请求形式为 `BASE_URL/v3/ai/...`。

**使用 Tiptap Cloud 文档（推荐）** — 在请求体中包含带有 `documentId` 的 `experimental_documentOptions`。服务器将自动从 Tiptap Document Server 获取文档、处理文档，并保存回修改。此方式要求你的 JWT 声明中包含 Document Server 凭据（见上文【Document Server 凭据】）。

工具端点仅接受一种文档来源：

- `document`: 内联的 Tiptap JSON
- `experimental_documentOptions`: 一个 Tiptap Cloud 文档引用

`experimental_documentOptions` 具有以下结构：

- `documentId` (`string`, required): Tiptap Cloud 文档标识符。
- `userId?` (`string | null`, optional): 归因于 AI 执行编辑操作的用户标识。
- `field?` (`string | null`, optional, default: `"default"`): 当省略或为 `null` 时，定位文档中的特定协作字段（Y.js XML 片段）。当你的文档在同一个 `documentId` 下存储多个可编辑字段时——例如标题和正文分开存储——请使用此项。

使用 Tiptap Cloud 文档的示例：

```json
{
  "experimental_documentOptions": {
    "documentId": "your-document-id",
    "userId": "ai-assistant"
  }
}
```

定位到非默认协作字段的示例：

```json
{
  "experimental_documentOptions": {
    "documentId": "your-document-id",
    "userId": "ai-assistant",
    "field": "title"
  }
}
```

使用内联文档的示例：

```json
{
  "document": {
    "type": "doc",
    "content": []
  }
}
```

<Callout title="线程需要云端文档" variant="info">
  线程和评论端点始终需要 `experimental_documentOptions.documentId`。
</Callout>

## 格式

大多数端点接受一个 `format` 字段：

- `"json"`: 标准 JSON 表示
- `"shorthand"`: Tiptap Shorthand，一种用于提示和模型输出的高效 token 格式

## Tool endpoints

### 获取工具定义

```txt
POST /v3/ai/toolkit/tools
```

返回可供 AI 代理使用的 [工具定义](/content-ai/capabilities/server-ai-toolkit/api-reference/tools)。

请求体：

- `editorContext` (`EditorContext`, required). 从 `getEditorContext` 函数获取的编辑器上下文。请参阅 [安装指南](/content-ai/capabilities/server-ai-toolkit/install)。
- `tools?` (`Record<string, boolean>`, optional): 启用或禁用单个工具。受支持的键请参阅
  [工具参考](/content-ai/capabilities/server-ai-toolkit/api-reference/tools#tool-availability)。
- `operationMeta?` (`string`, optional, default: `""`)
- `format?` (`'json' | 'shorthand'`, optional, default: `'json'`)
- `schemaAwarenessData?` (`SchemaAwarenessData`, optional, deprecated): 请改用 `editorContext`。为向后兼容而支持。

响应：

- `prompt` (`string`): 将其添加到你的 AI 请求的系统提示中。它会向 AI 说明 Tiptap 文档如何工作、可包含哪些元素，以及文档格式如何运作。
- `tools` (`array`)
  - `name` (`string`)
  - `description` (`string`)
  - `inputSchema` (`object`): AI 生成的工具 `input` 的 JSON schema。

### 执行一个工具

```txt
POST /v3/ai/toolkit/execute-tool
```

执行一个 Server AI Toolkit 工具，例如 `tiptapRead`、`tiptapEdit`、`getThreads` 或
`editThreads`。有关所有受支持工具及其类型形状，请参阅
[工具参考](/content-ai/capabilities/server-ai-toolkit/api-reference/tools)。

请求体：

- `toolName` (`string`, required)
- `input` (required): 由 AI 模型生成的工具输入。其结构取决于
  `toolName`；请参阅 [工具参考](/content-ai/capabilities/server-ai-toolkit/api-reference/tools)。
- `editorContext` (`EditorContext`, required unless `schemaAwarenessData` is provided)
- `schemaAwarenessData?` (`SchemaAwarenessData`, optional, deprecated): 请改用 `editorContext`。为向后兼容而支持。
- `toolConfig?` (optional): 由开发者提供、而非由 AI 模型提供的工具专属参数。请参阅
  [工具参考](/content-ai/capabilities/server-ai-toolkit/api-reference/tools)。
- `format?` (`'json' | 'shorthand'`, optional, default: `'json'`)
- `reviewOptions?` ([`ReviewOptions`](/content-ai/capabilities/server-ai-toolkit/api-reference/review-options), optional, default: `{ mode: 'disabled' }`)
- `experimental_commentsOptions?` (`{ threadData?: Record<string, unknown>, commentData?: Record<string, unknown> }`, optional): 在线程相关工具执行期间创建的新线程和评论所附加的元数据。
- exactly one of:
  - `document`
  - `experimental_documentOptions`

响应：

- `output`: 应由 AI 模型读取的工具响应。请参阅
  [工具参考](/content-ai/capabilities/server-ai-toolkit/api-reference/tools)。
- `toolResult`: 以供开发者解析的工具响应格式。不应由 AI 读取。请参阅
  [工具参考](/content-ai/capabilities/server-ai-toolkit/api-reference/tools)。
- `docChanged` (`boolean`): 文档是否因工具调用而发生变化
- `document` (`object | null`): AI 修改后的文档

### 流式执行工具

```txt
POST /v3/ai/toolkit/stream-tool
```

流式执行工具。请求结构与 `POST /v3/ai/toolkit/execute-tool` 相同。流式响应格式仍在最终确定中。

因为此端点通过实时连接在文档上工作，所以它可能返回 REST 端点不会返回的连接和并发错误。它们会作为响应流中的 `error` 事件到达 —— 参阅 [流式传输和会话错误](#streaming-and-session-errors)。

## 错误处理

REST 端点会返回以下格式的错误：

```json
{
  "error": {
    "message": "用人类可读的方式描述出错原因。",
    "status": 502,
    "code": "document_load_failed"
  }
}
```

- `message` (`string`): 人类可读的描述。不要基于它进行匹配——措辞可能会变化。
- `status` (`number`): HTTP 状态码，也会设置在响应中。
- `code` (`string`): 可用于分支判断的稳定标识符。验证失败会使用
  `code: "validation_failed"` 并添加一个 `issues` 数组。

该 API 使用标准 HTTP 错误码：

- `400 错误请求`：无效的请求体、无效的格式、缺少文档源，或缺少 Document 服务器凭据
- `401 未授权`：缺少或无效的 JWT 或应用程序 ID
- `404 未找到`：未知的端点或工具
- `422 无法处理的实体`：验证失败
- `429 请求过多`：重复请求或触发了速率限制保护
- `500 内部服务器错误`：意外的服务器错误
- `502 错误网关`：无法加载或保存 Tiptap Cloud 文档

### 流式传输和会话错误

[`POST /v3/ai/toolkit/stream-tool`](#stream-a-tool) 会在与 Tiptap Cloud 文档的实时连接上工作：它会实时加入该文档，因此 AI 会与其他协作者一起编辑最新内容。在建立或使用该连接时发生的错误，会以内联方式作为响应流中的 `error` 事件传递，而不是作为 HTTP 状态码：

```json
{ "type": "error", "code": "concurrent_edit_conflict", "status": 409, "message": "..." }
```

`status` 字段与对应的 HTTP 状态码一致。可能的 `code` 值如下：

| `code`                        | `status` | 发生时机                                                                                                                              | 处理方式                                                                                                                                                     |
| ----------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `concurrent_edit_conflict`    | 409      | 文档在工具运行期间发生了变化（例如，用户编辑了它），因此无法在过时内容之上应用更改。 | 重新运行请求。AI 会在下一次尝试中读取最新内容。可安全自动重试。参见 [处理并发编辑](#handle-concurrent-edits)。 |
| `schema_mismatch`             | 409      | 请求中的编辑器 schema 与该文档当前正在使用的 schema 不匹配，或者存储的文档无法用它解析。 | 为同一文档发送一致的 `editorContext` schema。不要使用相同的 schema 重试。                                                              |
| `websocket_auth_failed`       | 401      | 打开连接时，Tiptap Cloud 拒绝了文档凭据。                                                                 | 刷新 Document Server JWT 声明，然后重试。不要使用相同的令牌重试。                                                                           |
| `websocket_connection_failed` | 502      | 与 Tiptap Cloud 的连接在初始同步完成之前就关闭了。                                                                    | 使用退避策略重试。检查连接性和 Tiptap Cloud 状态。                                                                                                |
| `websocket_sync_timeout`      | 504      | 文档未能在时间限制内完成初始同步。                                                                          | 使用退避策略重试。大文档和较差的连接状况会使这种情况更容易发生。                                                                           |
| `session_limit_reached`       | 503      | 服务器正在处理的并发文档会话数量已达到上限。                                                                  | 使用退避策略重试。会话在空闲时会被释放。                                                                                                     |
| `session_creation_cancelled`  | 503      | 会话在完成连接之前就被终止，通常发生在服务器重启期间。                                                   | 重试。                                                                                                                                                         |

<Callout title="仅适用于 Tiptap Cloud 文档" variant="info">
  这些错误仅会发生在 `stream-tool` 使用的实时连接路径上。发送内联 `document` 的请求不会打开连接，因此不会返回会话或 WebSocket 错误。
</Callout>

### 处理并发编辑

在协作文档中，`concurrent_edit_conflict` 是预期内的。AI 在读取文档、调用模型并写入其更改时，用户可能会在此期间编辑同一文档。工具不会覆盖该编辑，而是拒绝写入，以免丢失任何工作内容。

请将其视为一个安全、可重试的信号：当你收到带有
`code: "concurrent_edit_conflict"` 的 `error` 事件时，请再次运行相同的工具调用。工具包会重新读取文档，
因此 AI 会基于用户的最新更改进行处理。为避免在被频繁编辑的文档上陷入长时间循环，请限制重试次数。

```ts
// `runStream` 发送 start/delta/end 消息，并解析为流的
// 最终 `error` 事件；如果流成功完成，则返回 `null`。
async function streamToolWithRetry(body: unknown, maxRetries = 3) {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    const error = await runStream(body)

    if (!error) {
      return // 成功完成
    }
    if (error.code !== 'concurrent_edit_conflict') {
      throw new Error(`stream-tool failed: ${error.code}`)
    }
    // 文档在传输过程中发生了变化——使用最新内容重试。
  }

  throw new Error('Tool call kept conflicting with concurrent edits.')
}
```
