## 这会做什么

DOCX 文件自带一个样式目录——即那些被命名的定义（`Heading1`、`Normal`、`Quote` 等），它们告诉 Word 正文文本、标题、超链接和列表应该如何呈现。DOCX 中的每个段落或运行（run）要么继承其中某一个命名样式，要么在行内覆盖单个属性。

没有此功能时，Tiptap 的 Convert API 会将这些样式“摊平”为每个节点的属性——你会看到字体大小、颜色和间距以行内方式应用到每个段落，但作为“事实来源”的、被 Word 用作基准的可复用命名样式会被丢弃。

该功能会把命名样式目录提取为一个**CSS 对象**（以 CSS 选择器为键），并且可以让你直接拿到该对象，或将其作为作用域 `<style>` 标签注入到你的页面中。结果：文档的基础排版通过 CSS 规则进行渲染（CSS 规则会层叠）——而不是对每个节点都使用行内属性——因此你的编辑器在视觉上能匹配原始 DOCX，而不会把文档模型“膨胀”。

## 安装

```bash
npm i @tiptap-pro/extension-import-docx@^0.10.0
```

## 它如何工作

一个四步流程：

1. **在导入请求中选择启用。** 扩展要求 Convert API 在 `cssStyles.enabled` 为 `true` 时包含样式目录。
2. **转换服务提取。** 转换器读取 DOCX 的样式定义，解析继承链（`basedOn`），将 OOXML 属性翻译为 CSS 等价物，并在导入响应中返回一个 `cssStyles` 对象。
3. **扩展构建 CSS 规则。** 将 `cssStyles` 对象编译为 CSS 字符串，并按你配置的选择器进行作用域隔离（默认 `.tiptap`）。
4. **自动注入或交给你处理。** 如果 `autoInject: true`，扩展会把作用域隔离的 `<style>` 元素追加到 `document.head`。如果 `autoInject: false`，你会在 `onImport` 回调中收到原始的 `cssStyles` 对象，并由你决定如何处理。

```ts
import { Import } from '@tiptap-pro/extension-import-docx'

Import.configure({
  appId: 'your-app-id',
  token: 'your-jwt',
  cssStyles: {
    enabled: true,      // 每次导入都提取样式
    autoInject: true,   // 自动以作用域 <style> 标签的形式注入
    selector: '.tiptap', // 将规则作用域限定在该选择器下
  },
})
```

或者在每次调用时控制：

```ts
editor
  .chain()
  .importDocx({
    file,
    cssStyles: { enabled: true, autoInject: false },
    onImport(context) {
      // `context.cssStyles` 是原始对象——你可以按自己喜欢的方式使用它
      console.log(context.cssStyles)
    },
  })
  .run()
```

## 何者可用

**选择器（16 个）：** 该功能只会把命名样式提取并映射到下列这些 CSS 选择器。

| 类型 | 选择器 |
|------|-----------|
| 块级 | `p`, `h1`–`h6`, `blockquote`, `ul li`, `ol li` |
| 行内 / 标记 | `strong`, `em`, `u`, `s`, `a`, `code` |

**CSS 属性（11 个）：** 上述每个选择器都可以接收这些属性的任意子集，具体取决于 DOCX 样式里定义了哪些。

| 属性 | DOCX 来源 | 备注 |
|----------|-------------|-------|
| `fontSize` | `w:sz`（半磅） | 转换为 `Npx`（`halfPoints / 2`） |
| `color` | `w:color` | Hex；跳过 `"auto"` |
| `fontFamily` | `w:rFonts` | 优先使用 `@w:ascii`，其次是 `hAnsi`，再其次是 `cs` |
| `fontWeight` | `w:b` | `"bold"` 或 `"normal"` |
| `fontStyle` | `w:i` | `"italic"` 或 `"normal"` |
| `textDecoration` | `w:u` + `w:strike` | 组合（例如 `"underline line-through"`） |
| `backgroundColor` | `w:shd[@fill]` | Hex；跳过 `"auto"` |
| `textAlign` | `w:jc` | `left` / `center` / `right` / `justify`（DOCX `both` → `justify`） |
| `marginTop` | `w:spacing[@before]` | Twips → `Npt` |
| `marginBottom` | `w:spacing[@after]` | Twips → `Npt` |
| `lineHeight` | `w:spacing[@line]` + `[@lineRule]` | `auto` → 无单位倍数；`exact`/`atLeast` → `Npt` |

**其他行为：**
- **继承链。** `basedOn` 会通过带环检测的迭代方式解析；子样式会包含它从父级继承来的所有内容。
- **本地化文档。** 使用非英文 Word 版本编写的 DOCX（例如西班牙语 `Título1`、德语 `Überschrift1`）会被映射回其规范名称（`heading 1`），从而仍能让选择器分配正常工作。
- **失败静默。** 无论出于任何原因（样式 xml 格式错误、出现了意外属性），提取过程抛出异常时都会返回 `{}`，并正常继续导入——该功能从不阻止一个可以正常工作的导入。

## 什么不会工作

<Callout variant="warning" title="这些是有意为之的限制">
这些并不是 bug。它们是该功能明确且已文档化的边界。如果你的工作流依赖下面任何一项，那么此功能并不是合适的工具。
</Callout>

- **伪类。** `:hover`、`:focus`、`:first-child` 等。DOCX 的样式模型没有对应概念；因此不会提取它们。
- **任意选择器。** 上表中的 16 个选择器就是完整映射。任何超出该映射用途的 DOCX 样式（`TableGrid`、`TOC1`、`FootnoteText`、自定义用户样式等）都会被丢弃。
- **字母间距。** 导出端编译器支持，但**导入时不会提取**。
- **选择器本身之外的特异性。** 注入的规则会被限定在你的选择器下（例如 `.tiptap h1 { … }`）。它们不会考虑与您自己的样式表之间的层叠顺序——你需要确保你的 CSS 不会意外地覆盖它们，或反过来。
- **行内覆盖。** 如果某个具体段落或运行在 Word 中对样式进行了行内覆盖（例如某个 `<w:r>` 直接设置了 `w:color`），这些覆盖会继续通过 Tiptap 节点上的节点属性进行渲染，而不是通过注入的样式表。你无法通过编辑注入的 CSS 来“取消”行内格式化的覆盖效果。
- **非文本样式。** 表样式（`w:tblStyle`）、编号样式（`w:numId`）、诸如页码这类字符样式——它们不属于该映射。

## 你可以期待

- 每次成功导入且 `autoInject: true` 时，会在 `document.head` 上挂载一个作用域隔离的单个 `<style>` 元素。后续导入会替换掉之前的元素——同一时间你只会有一个处于激活状态的。
- 在你的 `onImport` 回调上下文中返回原始的 `cssStyles` 对象，不论 `autoInject` 设置为何。你可以完全忽略自动注入，自行应用这些样式——把它们写入文件、发送给主题系统，等等，只要符合你应用的需求即可。
- 样式会限定在你的编辑器作用域中——默认是 `.tiptap`，可通过 `cssStyles.selector` 配置。只有当你把选择器设置成某个全局值（例如 `body`）时，才会“泄漏”到页面其他部分。
- 静默失败模式。若 DOCX 没有样式目录，或所有命名样式都落在 16-selector 映射之外，则 `cssStyles` 会为空，且不会注入任何样式标签。

## 不要期待

- **像素级复刻 Word。** Word 使用内置的布局引擎，其段落间距语义与 CSS 不同（例如折叠边距、段落前/后上下文相关的间距）。提取得到的 CSS 是对 DOCX **样式定义**的忠实翻译，而不是 Word 布局输出的重现。
- **在实时编辑时自动动态更新。** 样式只在每次导入时注入一次。如果用户之后在 Tiptap 中继续编辑，注入的样式不会重新计算——它们描述的是“导入文档的基础状态”，而不是当前编辑器状态。
- **跨文档合并。** 每次导入都会替换之前的 `<style>` 标签。连续导入两个 DOCX 文件只会得到第二个文档的样式。如果你需要合并，请从 `onImport` 中读取两个 `cssStyles` 对象并自行合并。
- **往返保证。** 提取再导出的循环不是无损的；关于相反方向的内容（以及它的注意事项），请参见导出页面 [CSS to DOCX](/conversion/export/docx/css-to-docx)。

## 配置参考

```ts
Import.configure({
  cssStyles: {
    enabled?: boolean,     // default: false. 打开提取。
    autoInject?: boolean,  // default: false. 将 <style> 标签追加到 document.head。
    selector?: string,     // default: '.tiptap'. 注入规则的作用域。
  },
})
```

按次调用覆盖：

```ts
editor.chain().importDocx({
  file,
  cssStyles: { enabled, autoInject, selector }, // 任意子集
  onImport(context) { /* context.cssStyles 在此处可用 */ },
})
```

REST API 查询参数（若你直接调用 `/import/docx`）：

```
POST /import/docx?extractCssStyles=true
```
